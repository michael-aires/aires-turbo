Identity

Authentication, actors, and tenant isolation

Actor types

User actors come from Better-Auth session cookies.

Agent actors come from short-lived signed JWTs minted by the core API.

Each request receives and echoes an x-request-id.

Organization resolution

User requests can provide x-organization-id. If omitted, a single-org user is auto-resolved.

Agent tokens carry org_id, so agent requests stay org-scoped without extra input.

Area Implemented behavior
Agent token TTL 15 minutes
Revocation check Validated against agent_token plus agent status on every call
Global wildcard tools:*
Category wildcard email:*, contract:*, and similar
Project scoping Resolved from tool input and token projectIds
Rate limiting Per-agent, per-tool, 60 requests per minute

Core service

REST API

GET /healthz

Health check. No auth.

GET /readyz

Readiness check. No auth.

ALL /api/auth/*

Delegated to Better-Auth for sessions, OAuth, JWT/JWKS, and related auth endpoints.

ALL /api/trpc/*

Shared tRPC router. Protected procedures require a session cookie.

GET /api/v1/whoami

Returns the resolved actor context plus request id.

{
  "actor": { "type": "agent", "agentId": "...", "scopes": ["email:send"] },
  "requestId": "..."
}
GET /api/v1/contacts?organizationId=<uuid>

Authenticated REST wrapper over contact.list. The REST adapter currently only exposes organizationId.

POST /api/v1/contacts

Authenticated REST wrapper over contact.create. Publishes contact.created on success.

{
  "organizationId": "org-uuid",
  "email": "buyer@example.com",
  "firstName": "Buyer",
  "lastName": "Name"
}
GET /api/v1/tools

Public tool catalog containing names, scopes, approval flags, categories, and cost tiers.

POST /api/v1/tools/:name

Canonical tool execution path. Applies org resolution, scope checks, project scoping, rate limits, input validation, and audit writes.

401 unauthenticated 403 scope or project denied 429 rate limited 500 handler error
POST /api/v1/agents/:agentId/tokens

User-only endpoint to mint a short-lived agent JWT for an owned active agent.

GET /events

Authenticated SSE stream. Events are filtered by organization and, for scoped agents, by project.

Streaming

Events, outbox, SSE, and webhooks

Event catalog

  • contact.created
  • contact.updated
  • activity.logged
  • email.sent
  • task.created
  • agent_run.started
  • agent_run.completed
  • approval.requested

Delivery chain

  1. Transactional insert into outbox_event
  2. Worker drains unpublished rows
  3. Push to Redis Stream aires:events
  4. Broadcast to Redis pub/sub for SSE
  5. Fan out to matching webhook subscriptions
Webhook property Behavior
Allowed protocol https in production, http allowed in local development
URL credentials Rejected
Private IP resolution Rejected
Headers x-aires-event, x-aires-signature, x-aires-delivery-id
Signature sha256=<hex-hmac> over the exact JSON body
Retry policy Exponential backoff, up to 8 attempts, then dead-letter

AI transport

MCP server

GET /sse

Requires an agent JWT. Opens an MCP SSE session.

POST /message?sessionId=<id>

Receives follow-up MCP client messages for the active session.

The MCP layer exposes tools as generic objects for transport. Actual authorization, validation, and audit behavior still happens in apps/core.

Channel extension

Chat SDK and WhatsApp integration

Status: this is the recommended next integration layer, not a currently running aires-turbo service. It also supersedes older Twilio-shaped WhatsApp notes elsewhere in the repo. For a Chat SDK rollout, the documented target should be the official Meta Cloud adapter at chat-sdk.dev/adapters/whatsapp.

Recommended packages

  • chat
  • @chat-adapter/whatsapp
  • @chat-adapter/state-redis
  • @chat-adapter/state-pg for a PostgreSQL-backed variant

This fits the current backend because Redis is already a core dependency and apps/core already centralizes auth, audit, and tool execution.

Documented WhatsApp behavior

  • DMs supported
  • reactions supported
  • cards partially supported
  • modals not supported
  • streaming is buffered
  • message history fetch is not supported

Aires therefore needs first-party CRM persistence for every inbound and outbound WhatsApp turn.

APP apps/chat-bots

Recommended new process for Chat SDK adapter setup, webhook handling, Redis-backed thread state, contact resolution, and channel routing. 

import { Chat } from "chat";
import { createWhatsAppAdapter } from "@chat-adapter/whatsapp";
import { createRedisState } from "@chat-adapter/state-redis";

const bot = new Chat({
  userName: process.env.WHATSAPP_BOT_USERNAME ?? "aires-sales",
  adapters: { whatsapp: createWhatsAppAdapter() },
  state: createRedisState(),
});

Sales rep WhatsApp channel

buyer-facing

This is the external buyer or prospect thread. One WhatsApp thread maps to one CRM conversation, owned by the assigned rep.

  • lead qualification
  • project FAQ replies
  • appointment scheduling
  • rep takeover from AI
  • manager escalation

Recommended CRM writes: create or resolve the contact, persist the message, write communication.provider = "whatsapp", and extend activity.kind with a first-class whatsapp value.

Sales manager WhatsApp channel

operational

This is a manager-specific oversight lane for approvals, reassignment, SLA alerts, and digest summaries.

  • approve or reject escalations
  • reassign hot leads
  • pause or resume automation
  • receive stale-thread alerts
  • receive daily summary digests

Recommended shape: one-to-one manager direct threads. That is the safer documented model because the official adapter surface explicitly documents DM support and does not document WhatsApp group-thread workflows.

UX constraints from WhatsApp

  • no edit after send
  • no delete after send
  • no modal workflows
  • streaming is buffered, not token-live
  • 3 or fewer buttons works best
  • use list messages for longer action sets
  • persist message history in Aires because Cloud API history fetch is unavailable

Recommended future scopes and tables

Scopes: chat:read, chat:reply, chat:assign, chat:escalate, chat:approve.

Recommended future tables: channel_account, channel_thread, channel_message, channel_assignment, channel_handoff, channel_digest_subscription.
